# Phase 10 — UI Design Contract

> Visual and interaction contract for Phase 10 (Authentication & Security Core).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

This phase introduces 6 net-new user-facing routes plus a header user menu to a
**frontend-complete bilingual EN/AR Saudi streetwear** ecommerce app. The contract
keeps surface area intentionally small. Auth UI must look and behave like the
existing checkout — same editorial tokens, same form pattern, same dignified copy.

## Spacing Scale

The project does not declare an explicit spacing token table — it consumes
Tailwind's default scale. Phase 10 binds the following values from that scale.
All values are multiples of 4. **All directional spacing MUST use logical
properties (`ms-*`, `me-*`, `ps-*`, `pe-*`, `space-x-reverse`-aware utilities).**

> **Scale source note:** The scale is bound from the Tailwind v4 default spacing
> scale; values outside the canonical {4, 8, 16, 24, 32, 48, 64} set
> (e.g., 12 px, 96 px) are accepted because the project policy is to use
> Tailwind defaults rather than a custom curated scale.

| Token | Tailwind | Value | Phase 10 Usage |
|-------|----------|-------|----------------|
| xs    | `2`      | 8 px  | Inline gap between input and helper text; chip padding; checkbox-to-label gap |
| sm    | `3`      | 12 px | Vertical gap inside a stacked field; label-to-input margin (label `mb-2` = 8px is also valid) |
| md    | `4`      | 16 px | Default gap between form rows on mobile; status-pill horizontal padding (`px-3 py-1`) |
| lg    | `6`      | 24 px | Page horizontal gutter (`px-6`); section-internal vertical rhythm |
| xl    | `8`      | 32 px | Card inner padding (`p-8` matches confirmation card); button horizontal padding (`px-10`) |
| 2xl   | `12`     | 48 px | Vertical rhythm between page hero and form (`space-y-12`); column gap on 2-col forms (`gap-x-12`) |
| 3xl   | `24`     | 96 px | Top/bottom padding on auth viewport (`py-24` matches confirmation page) |

**Editorial-form exceptions (carry over from `src/components/checkout/ShippingForm.tsx`):**

- Field vertical gap on multi-column forms: `gap-y-10` (40 px) — already established in checkout.
- Label `mb-2` (8 px) — established.
- Input vertical padding: `py-3` (12 px).
- Card pads: `p-8` (32 px) — matches `bg-[#111111]` card on confirmation page.
- **Touch target floor: 44 px on every interactive element.** Already enforced
  globally in `src/app/globals.css` (`a, button, [role="button"] { min-height: 44px; min-width: 44px; }`).
  Forms must respect this — input `min-h-[44px]` is mandatory (already in
  checkout pattern).

## Typography

Bind these exact values. Do NOT introduce new sizes or weights.
`tracking-tighter` (-0.025em) and the global `letter-spacing: -0.02em` on
`h1..h6` are already established in `globals.css`.

**The scale is exactly 4 distinct font sizes: `12 / 14 / 16 / 36–60 (responsive)`.**
Tier 1 is a single role with two breakpoints (`text-4xl md:text-6xl`) and is
counted as ONE size, not two. Reviewers must not double-count.

| Tier | Role(s) | Tailwind | Size (px) | Weight | Line Height | Letter-spacing | Used For |
|------|---------|----------|-----------|--------|-------------|----------------|----------|
| **1** | Display (responsive H1, also reused for sub-section heading on `/account`) | `text-4xl md:text-6xl font-bold uppercase tracking-tighter` | **36 / 60** (one role, two breakpoints) | 700 | 1.0 (`leading-none`) | -0.02em (global) | Page H1 on `/login`, `/signup`, `/forgot-password`, `/reset-password`, `/account`; "Recent orders" sub-section heading on `/account` (responsive bump retained) |
| **2** | Input value (mandatory anti-iOS-zoom) | `text-base` | **16** | 400 | 1.5 | 0 | All `<input>` and `<select>` text content. **Required by iOS Safari** — anything below 16 px triggers focus-zoom. |
| **3** | Body / Button label / Link / Helper-prose | `text-sm` (latin auto-bumped to 17 px in AR via `:lang(ar)` optical bump) | **14** | 400 (body, link, helper) / 700 (button label, uppercase tracking-widest) | 1.5 (body, link) / 1 (button) | 0 (body, link) / 0.1em (button) | Body prose under hero, "Don't have an account?" line, "Forgot password?" link, all primary CTA labels (`Sign in`, `Create account`, `Send recovery link`, `Update password`), "Sign in instead", "Continue shopping", date-line on `OrderRow`, total amount on `OrderRow`, signed-in-as line, dropdown menu items |
| **4** | Meta / Label / Helper-micro / Error / Eyebrow | `text-[12px]` (or `text-xs` = 12 px) | **12** | 700 (label, eyebrow) / 400 (helper, error) | 1.4 (label, error) / 1.6 (helper) | 0.1em (`tracking-widest` for label & eyebrow) / 0 (helper, error) | Form labels, helper copy ("8 characters minimum", "We'll send a recovery link…"), `<p role="alert">` error messages, eyebrow above H1 ("Account", "New account", "Reset"), "Reference" label on order rows, "Signed in as {email}" microline in dropdown panel |

**Weights:** exactly **2** — `400` (regular) and `700` (bold). No `500`, `600`, no italics on auth surfaces.

**Why 10 px was dropped:** the previous draft had a 10 px tier for labels, eyebrows,
and dropdown microlines. 10 px risks **WCAG 1.4.4 (Resize text)** failures at
200 % zoom on small viewports and reads as too small for IBM Plex Arabic glyphs,
which need more vertical room than Latin to remain legible. **All former 10 px
roles are merged into Tier 4 (12 px).**

**Why 24 px was dropped:** the previous draft used `text-2xl` (24 px) for a
single sub-section heading on `/account` ("Recent orders"). The orders list is
the only place that heading appears; reusing the Tier 1 Display
(`text-4xl md:text-6xl`) at the same responsive bump keeps the page visually
unified and avoids introducing a 5th size for one surface.

**RTL reminder:** `tracking-widest` and `tracking-tighter` apply to LATIN text;
in AR, IBM Plex Arabic does not require letter-spacing (Arabic ligatures break).
Wrap latin-only display text (e.g., the literal word "ORKI", reference codes) in
`dir="ltr"` spans. **Reference/order codes are always LTR.** The 12 px Tier-4
tracking-widest classes (`tracking-widest` on labels and eyebrows) render as
plain 12 px in AR — the optical-bump rule in `globals.css` already lifts AR
body to ~17 px equivalent, but Tier-4 micro-text stays at 12 px in both locales
(eyebrows and labels are short and high-contrast, so AR readability is acceptable).

## Color

ORKI uses only black and white plus a single error tint. **No accent color.**
The "10% accent" slot in the 60/30/10 model is filled by **pure white on black**,
which doubles as primary action color.

| Role | Token | Value | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `bg-black` / `--color-dominant` | `#000000` | Page background on every auth route + `/account` |
| Secondary surface (30%) | `bg-[#111111]` / `--color-secondary-surface` | `#111111` | Form/card surfaces (login form container, order row card, user-menu dropdown panel) |
| Accent (10%) | `bg-white` / `text-white` / `--color-accent-orki` | `#FFFFFF` | Primary CTA fill, primary CTA text-on-white-bg, focused border, body text on dark, hero H1 |
| Hairline border | `border-white/10` to `border-white/20` (`--color-orki-border` ≈ rgba(255,255,255,0.12)) | rgba(255,255,255,0.10–0.20) | Card borders, input bottom borders (default state), divider in dropdown |
| Muted text | `text-white/60` | rgba(255,255,255,0.60) | Helper text, body prose under hero |
| Subdued text | `text-white/40` | rgba(255,255,255,0.40) | Eyebrows, "Don't have an account?", inactive metadata |
| Decorative-only text | `text-white/20` | rgba(255,255,255,0.20) | Footer micro-copy ("Need help? Contact support") |
| Placeholder text | `placeholder:text-white/10` | rgba(255,255,255,0.10) | Input placeholder (matches checkout) |
| Destructive / Error | `text-red-400` | `#F87171` (Tailwind red-400) | Error message text under invalid field; **NEVER** as a fill, **NEVER** on borders. Aria-invalid borders use `border-white` (focus state), not red. |
| Focus ring (visible) | `focus:border-white` (form inputs use border-color shift), `focus-visible:ring-3 focus-visible:ring-ring/50` (buttons) | `#FFFFFF` border / `--ring` ring | All focusable elements |

**Accent reserved for:** Primary submit button fill (`bg-white text-black`), focus
border on inputs, the H1 itself, and the user-dropdown trigger underline-on-hover.
**Accent NEVER used for:** decorative dividers, status pills, info icons, secondary buttons.

**Status pill color (carry-over from `messages.Order.status`):**
Status pills are **outline-only**, no fill: `border border-white/30 text-white/80
text-[12px] uppercase tracking-widest px-3 py-1 rounded-none`. All six statuses
(`pending`, `confirmed`, `shipped`, `delivered`, `cancelled`, `refunded`) use
the same token — status conveyed by **label text**, not color. (Brand constraint
from CLAUDE.md: black & white only; no traffic-light color system.)

**Contrast verification (WCAG 2.1 AA):**

| Pair | Ratio | Pass |
|------|-------|------|
| `#FFFFFF` on `#000000` | 21:1 | AAA |
| `rgba(255,255,255,0.60)` on `#000000` | ~12.6:1 | AAA |
| `rgba(255,255,255,0.40)` on `#000000` | ~8.4:1 | AAA |
| `#F87171` (red-400) on `#000000` | ~5.4:1 | AA |
| `#000000` on `#FFFFFF` (CTA) | 21:1 | AAA |
| Eyebrow 12px white/40 on `#111111` | ~7.6:1 | AA (small text) |

`text-white/20` is decorative-only and may fall below AA — must NEVER carry meaning.

## Form Primitives

All auth forms (`LoginForm`, `SignupForm`, `ForgotPasswordForm`, `ResetPasswordForm`)
copy the **`ShippingForm` editorial pattern** verbatim. Single source of truth
lives at `src/components/checkout/ShippingForm.tsx` — do not invent a new form
chrome for auth.

### Input

```
className: 'w-full bg-transparent border-b border-white/20 py-3 min-h-[44px] text-white
           text-base placeholder:text-white/10
           focus:outline-none focus:border-white transition-colors duration-300 rounded-none
           aria-invalid:border-red-400/40'
```

| State | Visual |
|-------|--------|
| Default | Bottom border only, `border-white/20`, transparent fill |
| Focus | `border-white` (full opacity), no ring (the border IS the focus indicator); `outline-none` because border replaces it. Note: this is an **intentional editorial choice** — do not add a ring. |
| Invalid | `aria-invalid="true"` triggers `border-red-400/40` AND `<p role="alert">` below the field |
| Disabled | `opacity-50 cursor-not-allowed` |
| Loading (form submitting) | All inputs `disabled`, button shows spinner; do NOT add a per-input spinner |
| Autofill | iOS / Chrome may inject yellow background — neutralize with `[&:-webkit-autofill]:[box-shadow:inset_0_0_0_1000px_#000000] [&:-webkit-autofill]:[-webkit-text-fill-color:#FFFFFF]` |

**Required attributes on every input:**
- `id` (matched to label `htmlFor`)
- `aria-invalid="true"` when error present (omit when no error — never `false`, per existing `ariaProps()` helper)
- `aria-describedby="${id}-error"` when error present
- `autoComplete` (see table)
- `inputMode` for numeric / email keyboards
- `type` (`email`, `password`, `text`)
- `dir="ltr"` on email/password inputs **even in AR** — credentials are LTR by content; helper label remains RTL.

**autocomplete map (mandatory — improves both UX and password-manager support):**

| Field | autocomplete |
|-------|--------------|
| Login email | `email` (also `username`) |
| Login password | `current-password` |
| Signup email | `email` (also `username`) |
| Signup password | `new-password` |
| Forgot password email | `email` |
| Reset password new pwd | `new-password` |
| Reset password confirm | `new-password` |

### Label

```
className: 'block text-[12px] uppercase tracking-widest text-white/40 mb-2 font-bold'
```

`htmlFor` always set. AR localization keeps the same class — `tracking-widest` is
ignored by AR text rendering at this size (acceptable; `text-white/40` provides
the visual weight). Tier 4 (12 px) — note: previously drafted as 10 px; promoted
to 12 px to satisfy WCAG 1.4.4 and AR readability.

### Helper text

```
className: 'text-[12px] text-white/60 leading-relaxed mt-2'
```

Used for: "8 characters minimum", "We'll send a recovery link to this email",
"Make sure your password manager saves this".

`id="${id}-helper"` and `aria-describedby="${id}-helper"` (combine with error id
via space: `aria-describedby="${id}-helper ${id}-error"` when both present).

### Error

```
<p id="${id}-error" role="alert" className="text-red-400 text-xs mt-2">{message}</p>
```

(`text-xs` = 12 px in Tailwind v4 — Tier 4.)

**MessageKey-driven:** errors come back from Server Actions in the
`ActionResult<T>` envelope as `messageKey: 'Auth.errors.invalidCredentials'`,
which the form resolves via `useTranslations('Auth')`. Pattern matches
`src/app/actions/cart.ts` precedent. Generic copy ONLY — see Copywriting Contract.

### Checkbox (signup accept-terms only)

Use `@base-ui/react/checkbox` (already in dependency tree via shadcn base-nova).

```
Wrapper: 'flex items-start gap-3'
Box:     'mt-1 size-5 rounded-none border border-white/30
          data-[state=checked]:bg-white data-[state=checked]:border-white
          focus-visible:ring-3 focus-visible:ring-ring/50'
Indicator (check icon): lucide <Check className="size-4 text-black" />
Label:   'text-sm text-white/60 leading-relaxed cursor-pointer'
         (Inline links to /legal/terms and /legal/privacy use
          underline underline-offset-4 text-white)
```

`required` enforced both client-side (zod) and server-side (zod). The label MUST
contain BOTH `<Link>` to `/legal/terms` and `<Link>` to `/legal/privacy` (these
pages exist as of Phase 9). Bilingual interpolation via next-intl rich-text:

```ts
t.rich('acceptTerms', {
  terms: (chunks) => <Link href="/legal/terms" className="underline underline-offset-4">{chunks}</Link>,
  privacy: (chunks) => <Link href="/legal/privacy" className="underline underline-offset-4">{chunks}</Link>
})
```

### Button (primary / secondary / ghost)

| Variant | Class | Use |
|---------|-------|-----|
| Primary | `w-full sm:w-auto px-10 h-14 bg-white text-black rounded-lg font-bold uppercase tracking-widest text-sm flex items-center justify-center gap-2 hover:bg-white/90 transition-colors min-h-[44px] disabled:opacity-50` | Sign in, Create account, Send recovery link, Set new password |
| Secondary | `w-full sm:w-auto px-10 h-14 bg-transparent border border-white/30 text-white rounded-lg font-bold uppercase tracking-widest text-sm flex items-center justify-center gap-2 hover:border-white transition-colors min-h-[44px]` | "Cancel" on multi-step (rare in Phase 10) |
| Ghost / link | `text-sm font-bold uppercase tracking-widest text-white underline underline-offset-8 hover:text-white/80 transition-colors flex items-center gap-2 min-h-[44px]` | "Back to sign in", "Continue shopping" tertiary actions |
| Sign-out (in dropdown) | `w-full text-start px-4 py-3 text-sm text-red-400/90 hover:bg-white/5 transition-colors` | Sign out menu item only — red is allowed because action is destructive in spirit (clears session). |

**Loading state (primary):** swap label for inline `<Loader2 className="size-4 animate-spin" />` + truncated label, e.g. `Signing in...`. Disable button. Set `aria-busy="true"`. **Reduced-motion:** if `(prefers-reduced-motion: reduce)`, replace spinner with static dot `<span aria-hidden>•</span>` and rely on the disabled + label-text-change to convey state.

**Disabled vs. loading:** distinct. Disabled = invalid form (gray opacity). Loading = submitting (spinner + busy).

## Header Changes (Navbar)

## Accessibility (WCAG 2.1 AA) — non-negotiable

- **Landmarks:** every auth route has `<main>`. Navbar has `<header>` + `<nav aria-label="Main navigation">` (existing). Forms use `<form aria-label={t('heading')}>`.
- **Headings:** exactly one `<h1>` per page (the editorial display). No `<h2>` needed for single-card layouts.
- **Focus order:** Email → Password → (Forgot link) → Submit → (Alt link). For signup: Email → Password → Accept-terms checkbox → Submit. Tab order matches DOM order; do NOT use `tabindex` to reorder.
- **focus-visible:** all interactive elements show visible focus. Inputs use border-color shift to white. Buttons use shadcn's `focus-visible:ring-3 focus-visible:ring-ring/50`. Links use `focus-visible:underline underline-offset-8`.
- **Label associations:** every input has a `<label htmlFor>` (already in `Field` helper). NEVER use placeholder as label.
- **Error announcement:** errors use `role="alert"` (existing pattern). `aria-invalid="true"` and `aria-describedby` wired (existing `ariaProps()` helper).
- **Live region for form-level errors:** `<div role="alert" aria-live="polite">` above the submit button receives the messageKey-resolved error (e.g., `Auth.errors.invalidCredentials`).
- **Color contrast:** verified above (all ≥ AA).
- **Touch targets:** 44 × 44 enforced globally; honored on links inside dropdown via `min-h-[44px]`.
- **Keyboard:** Enter submits the form. Escape closes the user-menu dropdown (base-ui default). All actions reachable without a mouse.
- **Screen-reader-only descriptions:** the user-menu trigger has `aria-label="Account menu — signed in as {email}"` so the email is announced even though it's hidden in the trigger label.
- **autocomplete attributes:** mandatory map specified above (`email`, `username`, `current-password`, `new-password`).
- **Lang/dir:** set on `<html>` by `[locale]/layout.tsx` already; sub-elements inherit.
- **No autofocus:** do NOT auto-focus the email input. Users may have a screen reader scrolling the page; jumping to the input disorients. Let users tab in.
- **Type-size floor:** the smallest tier in the contract is 12 px (Tier 4). 10 px
  is forbidden because it can fail WCAG 1.4.4 (Resize text) at 200 % zoom on
  small viewports and is too small for IBM Plex Arabic glyphs.

## Anti-patterns (call out and reject)

The following must NOT appear in Phase 10 implementation. Checker should fail
the audit if any are present.

1. **Generic SaaS gradient blobs / hero illustrations** — auth pages are pure
   black with a single editorial card.
2. **Social-login buttons** ("Continue with Google", "Continue with Apple") —
   OAuth is explicitly deferred (CONTEXT.md). Do not render the buttons even
   as disabled placeholders.
3. **Password-strength meters with rainbow color bars** — there is no rainbow
   in the brand palette. If a password-strength signal is needed, it's a single
   helper line: "8 characters minimum" / "Strong password ✓" in `text-white/60`.
4. **Trust badges** ("256-bit SSL", "PCI compliant" lockup graphics) — unworthy
   of the brand. Trust is communicated by execution, not stickers.
5. **Marketing newsletter checkbox at signup** — Phase 10 scope is auth, not
   acquisition. No "Send me drops & releases" checkbox here. (Defer to a Phase 12
   marketing-consent workflow.)
6. **CAPTCHA badge** — Supabase rate-limits at the edge; no client-side captcha
   in Phase 10. (Deferred per RESEARCH §2.2.)
7. **"Welcome back, {firstName}!"** copy — too peppy for the brand voice. Use
   `Sign in` headline only.
8. **Avatar circles with user initials** — no avatar storage in Phase 10.
   `UserCircle` lucide icon is the only identifier in the trigger.
9. **Modal confirmation for sign-out** — friction without safety. Single-click
   sign-out is correct.
10. **Color-coded order status pills** (green=delivered, red=cancelled) —
    brand is black & white. All pills share one chrome.
11. **Email enumeration in errors** — never "User not found" or "Email already
    registered". Both wrong-email and wrong-password produce the same message.
    Signup with an existing email surfaces as `errors.unknown` plus an audit-log
    row (rather than disclosing the collision).
12. **Hover-only interactions** — UX-04 forbids it. Every action that reveals
    state on hover ALSO reveals it on focus.
13. **Locale-leaking class names** — never write `text-right` or `text-left` for
    directional content. Use `text-start` / `text-end` if alignment must be set
    (most cases inherit correctly from `dir`).
14. **`localStorage` for session/auth tokens** — SEC-01 forbids it. All session
    handling goes through `@supabase/ssr` httpOnly cookies. The auth UI must
    NEVER reference `window.localStorage` or `document.cookie`.
15. **10 px micro-text** — forbidden anywhere in Phase 10. The smallest tier is
    12 px (Tier 4). If a label, eyebrow, or helper feels too large at 12 px,
    the layout is wrong, not the type scale.
16. **A 5th font size** — the contract is bound to exactly 4 distinct sizes
    (`12 / 14 / 16 / 36–60 responsive`). Introducing `text-xl` (20 px),
    `text-2xl` (24 px), `text-3xl` (30 px), or `text-5xl` (48 px) on an auth
    or account surface is a contract violation.
