### Task 4.5: Build ResetPasswordForm with token-verify effect + recovery callback

<intent>

UI-SPEC §"Reset-password specific chrome" mandates the form not render until token is verified.

<approach>
**1. `src/app/api/auth/callback/route.ts`** (per 10-PATTERNS §"`src/app/api/auth/callback/route.ts`"):
```ts
import { NextResponse } from 'next/server';
import { createClient } from '@/lib/supabase/server';

/**
 * Supabase password-recovery redirect handler.
 * Receives ?code=...&next=... from the email link → exchanges for session
 * → redirects to /[locale]/reset-password.
 *
 * Inside /api so next-intl middleware excludes it (RESEARCH §7 #14).
 * Creates its own SSR client for the cookie write.
 */
export async function GET(request: Request) {
  const { searchParams, origin } = new URL(request.url);
  const code = searchParams.get('code');
  const next = searchParams.get('next') ?? '/en/reset-password';

  if (code) {
    const supabase = await createClient();
    const { error } = await supabase.auth.exchangeCodeForSession(code);
    if (!error) {
      return NextResponse.redirect(`${origin}${next}`);
    }
  }
  return NextResponse.redirect(`${origin}/en/forgot-password?error=invalid_link`);
}
```

**2. `src/components/auth/ResetPasswordForm.tsx`** (per 10-PATTERNS §"`src/app/[locale]/(auth)/reset-password/page.tsx`" Special concern + UI-SPEC §"Reset-password specific chrome"):

The page wrapping this is `/(auth)/reset-password/page.tsx` from Task 4.3, which renders the H1 chrome + this form. Because the recovery callback at `/api/auth/callback` already exchanges the code and sets the cookie BEFORE redirecting here, the form can rely on `getUser()` succeeding. But UI-SPEC also wants a "verifying" loading state in case the user navigates here directly. Strategy:

- Component is `'use client'`.
- On mount: call `createClient()` from `@/lib/supabase/client` and `supabase.auth.getUser()`. While pending: render the loading spinner per UI-SPEC §"Reset-password specific chrome":
  ```tsx
  <div className="text-center space-y-6">
    <Loader2 className="size-6 text-white/40 animate-spin mx-auto" aria-hidden />
    <p className="text-sm text-white/60">{t('verifyingLink')}</p>
  </div>
  ```
- If `getUser()` returns `null` user (no recovery session): render `BrandedErrorPage variant="error"` with `heading: t('linkExpiredHeading')`, `body: t('linkExpiredBody')`, `ctaLabel: t('requestNewLink')`, `ctaHref: '/forgot-password'`.
- If `getUser()` returns a user: render the form (two `type=password` inputs `new-password` + `new-password`, dir="ltr", autoComplete="new-password" on both, zod via `setPasswordSchema`).
- On submit success: render inline success `<p>{t('success')}</p>` + `<Link href="/login">{t('back')}</Link>`. (Or redirect to `/login` after 2 s — both acceptable; static success is simpler.)

Imports:
```tsx

```

CRITICAL: Do NOT call `setPasswordAction` until the user has a recovery session. The action itself protects via the SSR client (no session = `updateUser` fails); the loading-state-then-render pattern avoids flashing the form to a user who shouldn't see it.
</approach>
